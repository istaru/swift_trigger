# 领域文档

各工程技能在探索代码库时，应按以下规则读取本仓库的领域文档。

## 探索前先阅读

- 仓库根目录的 **`CONTEXT.md`**；
- 若根目录存在 **`CONTEXT-MAP.md`**，则优先读它——它会指向各上下文各自的 `CONTEXT.md`，读取与当前主题相关的部分。
- **`docs/adr/`** — 阅读与当前工作区域相关的 ADR。多上下文仓库还需检查 `src/<上下文>/docs/adr/`。

若上述文件不存在，**静默跳过，不必提示**，也不要主动建议创建。`/domain-modeling` 技能（通过 `/grill-with-docs` 和 `/improve-codebase-architecture` 触发）会在术语或决策真正确定时懒创建这些文件。

## 文件结构

单一上下文仓库（本仓库）：

```
/
├── CONTEXT.md
├── docs/adr/
│   ├── 0001-xxx.md
│   └── 0002-xxx.md
└── Sources/
```

## 使用术语表中的词汇

在议题标题、重构提案、假设、测试名称等输出中，使用 `CONTEXT.md` 术语表定义的词汇，不得改用术语表明确排除的近义词。

若需要的概念尚不在术语表中，这是一个信号——要么你在使用项目中不存在的语言（请重新考虑），要么术语表存在真正的空缺（记录下来，留给 `/domain-modeling` 处理）。

## 标记 ADR 冲突

若输出内容与已有 ADR 相矛盾，请明确指出，而非静默覆盖：

> _与 ADR-0007（事件溯源订单）相矛盾——但值得重新讨论，因为……_
